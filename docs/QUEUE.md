# Queues

## What works

- SASL `MSSBCBS` + CBS token auth (`$cbs` request/response link pair).
- Send to a queue. The broker advertises `MaxMessageSize` on attach so the SDK's batch APIs work. Both `SendMessagesAsync(IEnumerable<>)` and `ServiceBusMessageBatch` (which the SDK packs into a single AMQP transfer with `message-format=0x80013700` and a body of multiple `Data` sections) are unwrapped server-side and fanned out into individual enqueues — applies to queue sends and topic sends both.
- Receive from a queue. Backed by an in-memory store with async-blocking dequeue (TCS waiters, no spin).
- Multiple queues. Any address auto-creates an `InMemoryQueue` via `QueueStore`.
- PeekLock receive with `CompleteMessageAsync` / `AbandonMessageAsync` / `DeadLetterMessageAsync` (including reason/description).
- Lock expiry timer: peek-locked messages held past `LockDuration` are re-enqueued with bumped delivery count.
- `MaxDeliveryCount` enforcement → automatic DLQ routing on exceeded count.
- `$DeadLetterQueue` sub-address routing for explicit and automatic dead-lettering.
- `$management` link, registered per-queue at `<name>/$management` and per-DLQ at `<name>/$DeadLetterQueue/$management` (DLQ management only handles `peek-message` and `renew-lock`; schedule/session/defer ops return 501 there), with `com.microsoft:renew-lock` (`RenewMessageLockAsync`), `com.microsoft:peek-message` (`PeekMessageAsync` / `PeekMessagesAsync`), `com.microsoft:schedule-message` (`ScheduleMessageAsync`), `com.microsoft:cancel-scheduled-message` (`CancelScheduledMessageAsync`), `com.microsoft:renew-session-lock` (`RenewSessionLockAsync`), `com.microsoft:get-session-state` (`GetSessionStateAsync`), `com.microsoft:set-session-state` (`SetSessionStateAsync`), `com.microsoft:receive-by-sequence-number` (`ReceiveDeferredMessageAsync` / `ReceiveDeferredMessagesAsync`), and `com.microsoft:update-disposition` (Complete/Abandon/DeadLetter on a deferred message).
- Stable per-message sequence numbers (`x-opt-sequence-number`) assigned on first enqueue and preserved across redeliveries; primary and DLQ buffers each maintain their own sequence space.
- Topics + subscriptions. `CreateTopic(name, TopicOptions)` declares a topic with a fixed set of subscriptions; senders attach to the topic root and each delivery is fanned out (deep-cloned via AMQP encode/decode) into every subscription's `InMemoryQueue`. Receivers attach to `topic/Subscriptions/sub` and get the full queue surface — peek-lock, abandon, dead-letter, lock renewal, scheduled delivery, sessions, and `topic/Subscriptions/sub/$management`. Receiver attach to a bare topic address is rejected with `amqp:not-allowed`. Topics and subscriptions must be pre-declared (no auto-create) — matches Azure semantics.
- Subscription rules — correlation and SQL filters. `SubscriptionOptions` accepts a list of `RuleFilter`s; `CorrelationRuleFilter` matches on system properties (`CorrelationId`, `MessageId`, `To`, `ReplyTo`, `Label`/Subject, `SessionId`/GroupId, `ReplyToSessionId`/ReplyToGroupId, `ContentType`) and arbitrary `ApplicationProperties` entries (AND across all set fields). `SqlRuleFilter` runs a hand-rolled recursive-descent parser/evaluator over the Service Bus SQL dialect: string/int/double/bool/null literals, comparisons (`=`, `<>`/`!=`, `<`/`<=`/`>`/`>=`), `AND`/`OR`/`NOT` with SQL92 Kleene three-valued logic, `IS [NOT] NULL`, `[NOT] LIKE` with `%`/`_` wildcards and `ESCAPE`, `[NOT] IN (...)`, `EXISTS(prop)`, system-property access via `sys.<Name>`, case-insensitive keywords and identifiers. Arithmetic is not implemented (rare in real filters; add if needed). Multiple rules on a subscription are OR-combined. A subscription with no rules in `SubscriptionOptions` is seeded with `$Default` (match-all); subsequent rules are auto-named `rule1`, `rule2`, etc.
- Runtime subscription rule management. `ServiceBusClient.CreateRuleManager(topic, sub)` operates on the subscription's `$management` link with `com.microsoft:add-rule` (`CreateRuleAsync`), `com.microsoft:remove-rule` (`DeleteRuleAsync`), and `com.microsoft:enumerate-rules` (`GetRulesAsync`). Rules are stored thread-safely per subscription. AddRule decodes the SDK's plain-Map filter encoding (`sql-filter` / `correlation-filter` keys); EnumerateRules emits the response body as a Map containing AMQP described-list values for each rule with the standard descriptors (`com.microsoft:rule-description:list`, `:sql-filter:list`, `:correlation-filter:list`, `:true-filter:list`, `:empty-rule-action:list`). Descriptors are emitted symbolically — Microsoft.Azure.Amqp's generic-Map decoder mishandles ULong-form descriptors at this nesting depth. Removing every rule (including `$Default`) drops all messages on that subscription, matching Azure semantics.
- Message TTL. `QueueOptions.DefaultMessageTimeToLive` and the per-message `ServiceBusMessage.TimeToLive` (AMQP `Header.Ttl`) combine via `min` into an absolute expiry stamped on `Properties.AbsoluteExpiryTime`. A per-queue `ExpiryStore` (single timer aimed at the soonest expiry) evicts idle messages from the ready buffer when expiry fires; peek-locked messages are deferred and DLQ'd on lock-release via the existing reroute path. With `DeadLetteringOnMessageExpiration = true` the message lands on the DLQ with `DeadLetterReason = "TTLExpiredException"`; with the flag false it's silently dropped (matching Azure's default-off behaviour).
- Sessions. Messages with `Properties.GroupId` set route into per-session sub-buffers that share the queue's sequence-number space. Receivers attaching with the `com.microsoft:session-filter` filter acquire a session lock (only one receiver per session at a time); the response attach echoes the session-id and stamps `com.microsoft:locked-until-utc` so the SDK knows when to renew. A null filter value picks the next available unlocked session — the attach is parked until a session with pending messages becomes available (via enqueue or another holder releasing) or the SDK's `com.microsoft:timeout` (default 60s) elapses, matching real Service Bus's blocking accept-next behaviour. A contended attach is rejected with `com.microsoft:session-cannot-be-locked` so the SDK surfaces it as `ServiceBusFailureReason.SessionCannotBeLocked`. The session lock is released when the receiver detaches, and a different connection can immediately re-acquire. All queues support sessions implicitly — `RequiresSession` exists in `QueueOptions` only for parity with Azure's `CreateQueueOptions`.
- Single-entity AMQP transactions. The server registers a `Coordinator` link target via `QueueLinkProcessor` (with an `AddressResolver` shim because `ContainerHost` casts `attach.Target` to `Target` before delegating to the link processor) and a shared `TxnManager` decodes `Declare` (returning `Declared{TxnId}`) and `Discharge`. Sends and dispositions arriving with a `TransactionalState` delivery state are staged as closures against the txn instead of applied immediately — on `Discharge{fail=false}` the staged enqueues fan out to the queue/topic and staged settlements apply via `Complete`/`Abandon`/`Reject`; on `Discharge{fail=true}` staged ops are dropped, so in-flight peek-locked messages stay locked and redeliver on lock expiry (matching Service Bus's at-least-once contract). Cross-entity "send-via" transactions aren't implemented — `ServiceBusTransaction` against a single sender or single receiver+sender on one entity works.
- Receiver-detach lock semantics match Azure. AMQP `released` disposition (which AMQPNetLite emits for unsettled deliveries when a link closes) is treated as "lock held until expiry" rather than as an immediate abandon — matching Service Bus's at-least-once contract. Explicit `AbandonMessageAsync` (which the SDK sends as `modified` with `delivery-failed=true`) still requeues immediately as before.
- Duplicate detection. `QueueOptions.RequiresDuplicateDetection` + `DuplicateDetectionHistoryTimeWindow` (defaults to 1 minute when unset) wires a sliding-window `MessageId` set into `InMemoryQueue.Enqueue`; a second send carrying the same `Properties.MessageId` within the window is silently dropped (broker still returns Accepted to the sender, matching Azure's "client never sees an error" semantics). Messages without a `MessageId` aren't dedup-eligible. `TopicOptions` exposes the same pair and runs dedup at the topic ingress before fan-out, mirroring Azure's "dedup is a topic-level property" model — subscriptions never see more than one copy of a duplicate. Eviction is lazy on each add (no timer), so there's no extra teardown surface. Standard+ tier on the Azure side; the parity test skips on Basic.
- Message deferral. `DeferMessageAsync` (SDK sends `Modified{UndeliverableHere=true}` on the receive link) drops the message out of the ready buffer into a per-queue `DeferredStore` indexed by sequence number. `ReceiveDeferredMessageAsync(sequenceNumber)` / `ReceiveDeferredMessagesAsync(IEnumerable<long>)` go through `$management` (`com.microsoft:receive-by-sequence-number`) — each retrieval issues a fresh lock-token + lock-expiry timer; lock expiry leaves the message in the deferred store (deferred messages don't time out, matching Azure). Subsequent `CompleteMessageAsync` / `AbandonMessageAsync` / `DeadLetterMessageAsync` on the deferred message route through `$management` (`com.microsoft:update-disposition`) with `disposition-status` `completed` / `abandoned` / `suspended` (DLQ). Deferred messages aren't visible to normal receivers and aren't currently surfaced in `Peek` (parity gap; Azure includes them). Note: this also fixed an unrelated bug in our `Modified{UndeliverableHere=true}` handling — we previously routed it to DLQ, but the real Service Bus DLQ disposition is `Rejected` with `com.microsoft:dead-letter` error condition (already handled separately).
- Auto-forwarding. `QueueOptions.ForwardTo` redirects sends to a different queue or topic at ingress (after dedup; before sequence-number assignment on the source) — the source never buffers the message, so receivers attached to the source see nothing. `ForwardDeadLetteredMessagesTo` redirects dead-letters (explicit, max-delivery-count, TTL expiry) to the target instead of the source's `$DeadLetterQueue` buffer; reason/description annotations are preserved. Subscriptions inherit both knobs through their `SubscriptionOptions.Queue`. Targets are resolved by name against the live `QueueStore` at delivery time; an unknown forward target dead-letters the message on the source with reason `ForwardingTargetNotFound`. Forward chain depth limits aren't enforced (Azure caps at 4 hops).
- `AmqpServer` is `IAsyncDisposable`. `DisposeAsync` closes the `ContainerHost` (releasing the listener port) and disposes the queue/topic graph — `ScheduledStore` flush timers, `MessageBuffer` lock timers, parked accept-next-session waiters. Idempotent and safe to call before `Start`. Caller contract: dispose every `ServiceBusClient` first (an active client disconnect during teardown can race AMQPNetLite's link-close handler — same lock-ordering hazard documented in `DeleteQueue`). Constructor takes an optional listener port so tests can stand up isolated servers; defaults to 5672. The xUnit harness still keeps `LocalServer` as a process-singleton on 5672 for speed; `ShutdownTests` exercise the dispose path on 5673.

## Known limitations

- No auth check beyond accepting the CBS `put-token` shape.
- SQL filter arithmetic (`+`, `-`, `*`, `/`, `%`) is not implemented. The grammar covers the operators that show up in real subscription rules; arithmetic can be added if a use case appears.
- `DeleteQueue` removes the queue from `QueueStore` but intentionally does not unregister the per-queue `$management` request processor on `ContainerHost` — that path deadlocks against AMQPNetLite's link-close handler when a client disconnect races teardown. Test queue names are unique GUIDs so re-registration can't collide; the leak is one dictionary slot per delete for the lifetime of the server. See `docs/DELETE_QUEUE_DEADLOCK.md` for the full thread trace.
- No topic-root `$management` link — only per-subscription. Cross-subscription peek isn't supported.
- Azure parity for sessions and topics requires a Standard-tier namespace. The default `task sb:up` provisions Basic; run `task sb:up:standard` instead to enable `SessionTests` and `TopicTests` against the real namespace (~$10/mo base — tear down with `task sb:down` when finished).

## Next steps (rough priority order)

### Likely to bite real apps

1. **ReceiveAndDelete mode.** Only peek-lock is exercised today. ReceiveAndDelete attaches with `settle-mode = pre-settled` (sender-settled), so the broker must drop the message on transfer instead of waiting for a disposition. Probably works via AMQPNetLite's settlement plumbing but needs a parity test to confirm — and to confirm DLQ-on-expiry / max-delivery semantics still apply correctly when the message is never locked.
2. **SQL rule actions.** `SqlRuleAction` lets a subscription rule mutate the matched message (`SET user.foo = 'bar'`, `REMOVE sys.Label`, etc.) before fan-out into the subscription buffer. Filters are in; the action half of `RuleDescription` isn't. Parser already handles the expression grammar, so this is mostly plumbing the SET/REMOVE statements through `AddRule` decode and applying them in the topic fan-out path.

### Plausible but lower priority

3. **Cross-entity (send-via) transactions.** Single-entity txns are in; `via-partition-key` routing for `ServiceBusClient.CreateTransactionalSender` against a different "via" entity isn't.
4. **Transfer dead-letter queue.** Azure separates "this message couldn't be auto-forwarded" failures onto a `$Transfer/$DeadLetterQueue` sub-buffer rather than the normal DLQ. Today an unknown forward target dead-letters on the source's regular DLQ with `ForwardingTargetNotFound` — reasonable simplification, but not Azure-accurate.
5. **`GetQueueRuntimePropertiesAsync` / message counts.** `ActiveMessageCount`, `DeadLetterMessageCount`, `ScheduledMessageCount`, `TransferMessageCount`, `TransferDeadLetterMessageCount`. The SDK's admin client hits HTTP for these — see HTTP/UI plane below.
6. **`MaxSizeInMegabytes` quota enforcement.** Sends past the configured size cap should fail with `QuotaExceededException`. Not enforced.
7. **`AutoDeleteOnIdle`.** Entity self-deletion after an idle window. Niche.
8. **Prefetch.** `ServiceBusReceiverOptions.PrefetchCount`. AMQPNetLite handles AMQP credit-flow so this likely just works, but worth a sanity test under high prefetch values.
9. **SQL filter arithmetic.** Only if a real subscription rule needs it.
10. **Deferred messages in `Peek`.** Real Service Bus surfaces deferred messages in peek output; we hide them. Cheap to add — include the `DeferredStore` snapshot in `MessageBuffer.Peek`'s result.

### HTTP / UI plane

11. **HTTP management plane.** `ServiceBusAdministrationClient` (queue/topic/subscription/rule CRUD, `GetRuntimeProperties`) talks to a REST surface at `https://<namespace>.servicebus.windows.net/<entity>` — *not* AMQP. Today the emulator only speaks AMQP, so admin-client code paths are unusable against `LocalServer`. Adding a Kestrel listener that responds to the Atom-XML management API would unlock `ServiceBusAdministrationClient`, the Azure portal's Service Bus Explorer protocol, and the standalone Service Bus Explorer tool. Endpoints in scope: `GET/PUT/DELETE /<entity>` (entity description Atom feed), `GET /<entity>/messageCountDetails`, `GET /$Resources/queues`, `GET /<topic>/Subscriptions`, `GET/PUT /<topic>/Subscriptions/<sub>/Rules`.
12. **Local UI.** Once the HTTP plane exists, a thin static dashboard (one HTML page served by the same Kestrel host) that lists queues/topics/subscriptions, shows per-entity message counts, peeks messages, and exposes a "send test message" form would make the emulator usable for ad-hoc poking without spinning up Azure-side tooling. Service Bus Explorer parity is the stretch goal; a minimal "what's in my queues right now" view is the MVP.
